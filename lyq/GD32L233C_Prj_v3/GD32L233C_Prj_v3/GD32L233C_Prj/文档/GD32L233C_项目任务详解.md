# GD32L233C 边缘自治电网监测系统

## 1. 系统概述

本项目基于 GD32L233C 微控制器（ARM Cortex-M23, **32KB RAM**, **256KB Flash**）实现电网三相电压/功率信号的采集、故障检测、数据分级存储及主控按需拉取上传的完整自治闭环系统。采用 FreeRTOS 多任务调度，通过 LoRa 无线模块（ATK-MWCC68D）与主控通信。

### 1.1 核心亮点

| 亮点 | 说明 |
|------|------|
| **主控时间戳轮询** | 不再定时推送，改为主控下发 `poll_ts` 时间戳 → 设备回溯查找对应时刻前后 1 周期数据上传。**按需拉取，零冗余传输**。 |
| **SRAM 环缓冲 + 故障快照 Flash** | 40 点 SRAM 环形缓冲（最新 2 周期） + 40KB Flash 故障快照存储（最多 5 条）。`poll_ts` 从 SRAM 环缓冲中逆向查找。 |
| **int16 定点数压缩存储** | `NodeSample_t` 全部字段 ×10000 转为 `int16_t`，每采样点 **52B**（24×int16 + uint32 timestamp）。 |
| **通用字段命名** | `pg1~3` / `qg1~3` / `vmag1~9` / `vangle1~9`，不绑定具体物理通道号，适配不同节点映射。 |
| **2 级 Flash 存储** | 波形数据（64KB 环写） | 故障快照（40KB，最多 5 条） |
| **10ms 滑动窗口故障检测** | 每 10 个采样点触发一次 RMS / 最大差分斜率(dv_dt) / 峰峰值(peak) 三特征综合判断，精准定位故障点位置（精确到窗口内采样点索引）。 |
| **DAC → 导线 → ADC 硬件在环录波** | DAC 输出正弦波经外接导线 → ADC 采集 → DMA 乒乓缓冲 → Flash 存储。预警 12kHz/250ms，紧急 15kHz/450ms。 |
| **电池电压监测** | ADC_CH10 (PC0) 软件触发，开机 4 秒采集 20 点计算基准值，后续每次上传节点头时附带当前电压百分比。 |

### 1.2 主要功能

- **三相电力数据采集**：1kHz 采样，50Hz 每周期 20 点，2 周期（40 点）SRAM 环形缓冲
- **仿真数据注入**：60 点正常 + 20 点故障交替循环，模拟真实电网波动
- **中频故障检测**：10ms / 10 点滑动窗口，实时检测电压暂升/暂降/过压/欠压
- **分级故障响应**：正常(NORMAL) → 预警(WARNING, 12kHz 录波 250ms) → 紧急(DANGER, 15kHz 录波 450ms + 备用电源切换)
- **2 级 Flash 存储**：波形录波数据(64KB) + 故障快照(40KB)
- **命令驱动上传**：主控下发请求 → 设备按需上传，支持时间戳轮询、故障波形检索、Flash 擦除
- **电池健康监测**：上位基准值标定 + 实时百分比上传

### 1.3 硬件资源分配

| 外设 | 用途 | 说明 |
|------|------|------|
| TIMER1 | ADC 波形采样触发 | PWM CH1 输出 → ADC 外部触发（EXTTRIG_REGULAR_T1_CH1） |
| TIMER2 | DAC 输出触发 | TRGO → DAC 触发，DMA_CH2 循环，256 点正弦 LUT |
| DAC (PA4) | 正弦波输出 | DMA_CH2 循环，51.2kHz 输出率 |
| ADC_CH0 (PA0) | 波形采集 | DMA_CH1，乒乓缓冲（2×64 点），连续模式 + 外部触发 |
| ADC_CH10 (PC0) | 电池电压采集 | 软件触发 + 单次转换，分压比 5:1 |
| DMA_CH1 | ADC 波形数据传输 | 半满/满中断触发 Flash 写入 |
| DMA_CH2 | DAC 数据传输 | 循环模式，自动输出正弦波 |
| USART1 (PA2/PA3) | LoRa 通信 | 115200bps，ATK-MWCC68D |
| USART0 (PA9/PA10) | 调试日志 | 115200bps，printf 重定向 |

### 1.4 Flash 存储布局

| 区域 | 起始地址 | 大小 | 用途 | 写入粒度 |
|------|----------|------|------|----------|
| 程序代码 | 0x08000000 | ~128KB | 固件 | - |
| 波形数据 | 0x08020000 | **64KB** | DMA 录波原始数据（环形覆盖） | 64 点/次 |
| 故障快照 | 0x08030000 | **40KB** | 故障时刻前后各 2 周期快照（最多 5 条） | 3132B/条 |

## 3. 数据结构（重点）

### 3.1 仿真数据点 — ZDataPoint_t

```c
typedef struct {
    float pg1; float pg2; float pg3;       // 有功功率 (3路)
    float qg1; float qg2; float qg3;       // 无功功率 (3路)
    float vmag1; float vmag2; float vmag3; // 电压幅值 (路1~3)
    float vmag4; float vmag5; float vmag6; // 电压幅值 (路4~6)
    float vmag7; float vmag8; float vmag9; // 电压幅值 (路7~9)
    float vangle1; float vangle2; float vangle3; // 电压相角 (路1~3)
    float vangle4; float vangle5; float vangle6; // 电压相角 (路4~6)
    float vangle7; float vangle8; float vangle9; // 电压相角 (路7~9)
} ZDataPoint_t;
```

仿真数据源包含 `g_zdata_normal[60]`（正常数据）和 `g_zdata_fault[20]`（故障数据），位于 `.rodata` 段。

### 3.2 节点采样点 — NodeSample_t（int16 定点数压缩）

```c
typedef struct {
    int16_t pg1;     int16_t pg2;     int16_t pg3;
    int16_t qg1;     int16_t qg2;     int16_t qg3;
    int16_t vmag1;   int16_t vmag2;   int16_t vmag3;
    int16_t vmag4;   int16_t vmag5;   int16_t vmag6;
    int16_t vmag7;   int16_t vmag8;   int16_t vmag9;
    int16_t vangle1; int16_t vangle2; int16_t vangle3;
    int16_t vangle4; int16_t vangle5; int16_t vangle6;
    int16_t vangle7; int16_t vangle8; int16_t vangle9;
    uint32_t timestamp;
} NodeSample_t;  // 每点 24×2 + 4 = 52B
```

> **设计意图**：9 路电压幅值 + 9 路相角 + 3 路有功 + 3 路无功 = 24 个 float。使用 `int16_t ×10000` 压缩存储，每采样点 52B。

### 3.3 环缓冲 — NodeBuffer_t

```c
typedef struct {
    NodeSample_t buffer[NODE_BUFFER_SIZE];  // 40 点 = 2 周期 = 2080B
    uint16_t write_index;                     // 当前写入位置 (环形)
    uint16_t cycle_count;                     // 累计采样点计数
} NodeBuffer_t;
```

> `cycle_count` 达到 20（1 周期）后才启动故障检测；环缓冲仅保留最新 2 周期数据在 SRAM 中。

### 3.4 故障快照记录 — PendingFault_t

```c
typedef struct {
    uint32_t fault_time;         // 故障发生的时间戳
    uint32_t flash_off;          // 快照在 NODE_FAULT_SNAP_ADDR 中的偏移
    uint16_t point_count;        // 实际故障点数
    uint8_t  severity;           // 故障级别 (SEVERITY_WARNING / DANGER)
    uint8_t  fault_type;         // 故障类型
    uint8_t  valid;              // 1=待上传
} PendingFault_t;
```

每条快照占 Flash 3132B = 12B Header(fault_time + severity + fault_type + point_count + padding) + 40 × 52B 原始 `NodeSample_t`。最多 5 条（MAX_PENDING_FAULTS=5）。

### 3.5 中频滑动窗口 — MidFreqWindow_t

```c
typedef struct {
    float voltage_mag[MID_FREQ_WINDOW_SAMPLES]; // 10 点 = 10ms
    uint16_t index;              // 窗口写入索引
    float rms;                   // 窗口 RMS（最近一次计算）
    float dv_dt;                 // 最大相邻差分斜率
    uint16_t dv_dt_index;        // 最大差分斜率发生的窗口内索引
    float peak;                  // 窗口峰峰值（max - min）
    float thd;                   // THD 总谐波失真（预留）
} MidFreqWindow_t;
```

> 使用 10 点 / 10ms 滑动窗口。当前使用 vmag1（第 1 路电压幅值）进行故障检测。

### 3.6 故障记录 — FaultRecord_t

```c
typedef struct {
    uint32_t timestamp;          // 故障检测时间戳
    FaultType_t fault_type;      // 故障类型（暂升/暂降/过压/欠压）
    SeverityLevel_t severity;    // 故障级别（警告/危险）
    float fault_value;           // 故障时刻 RMS 值
    uint32_t flash_offset;       // 对应波形在 Flash 中的起始偏移
    uint32_t sample_count;       // 波形采样点数
    uint16_t sample_rate;        // 波形采样率（12kHz/15kHz）
    uint8_t  valid;              // 有效标志（上传后置 0）
    uint8_t  node_index;         // 故障节点号
} FaultRecord_t;
```

最多 5 条（`MAX_FAULT_RECORDS=5`）。

### 3.7 传感器数据 — SensorData_t（任务间传递）

```c
typedef struct {
    ZDataPoint_t zdata;
    uint32_t timestamp;
} SensorData_t;
```

> 这是 `data_collect_task` → `g_sampledata_queue` → `data_judge_task` 的传递载体。`data_judge_task` 中 `node_sample_process()` 将 float 转为 `int16_t` 定点存入环缓冲。

### 3.8 节点头 — NodeUploadHeader_t

```c
typedef struct {
    uint8_t  data_type;         // DATA_TYPE_NODE_HEAD (0x01) 或 DATA_TYPE_FAULT_HEAD (0x07)
    uint8_t  severity;          // 故障级别
    uint8_t  fault_type;        // 故障类型 (FAULT_NONE=正常)
    uint8_t  fault_pending;     // 是否有待上传故障快照 (1=有)
    uint8_t  node_index;        // 节点号 (0~9)
    uint32_t timestamp;         // 时间戳 (ms)
    uint16_t sample_rate;       // 采样率 (1000Hz)
    uint16_t total_points;      // 后续 raw 数据总点数
    uint8_t  battery_pct;       // 电池电量百分比 (0~100, 255=未就绪)
    float    health_score;      // 健康度
} NodeUploadHeader_t;
```

## 4. 故障检测机制

### 4.1 检测流程

```
node_sample_process() 每 1ms 被调用一次
    │
    ├── 写入环缓冲 + cycle_count++
    │
    ├── cycle_count < 20 时跳过检测（不足 1 周期）
    │
    └── 每累计 10 个采样点 触发一次故障检测:
            │
            ├── 将最新 10 个 vmag1 点加载到 MidFreqWindow_t.voltage_mag[10]
            │
            ├── detect_mid_freq_fault() 计算:
            │   ├── RMS = sqrt(Σv² / 10)
            │   ├── Peak = max - min
            │   └── dv_dt = max(|v[i] - v[i-1]|), 记录 dv_dt_index
            │
            ├── health_calculate(rms, dv_dt, peak) → 更新综合健康度
            │
            └── 电压阈值判断:
                    RMS > 1.10pu → FAULT_VOLTAGE_SWELL  (暂升)
                    RMS > 1.05pu → FAULT_OVER_VOLTAGE    (过压)
                    RMS < 0.90pu → FAULT_VOLTAGE_SAG     (暂降)
                    RMS < 0.95pu → FAULT_UNDER_VOLTAGE   (欠压)
```

### 4.2 dv_dt 故障点精确定位

`dv_dt_index` 记录窗口内最大相邻差分斜率的位置。故障发生时，利用此索引反算出环缓冲中故障发生的精确位置：

```
fault_idx = (window_pos - 10 + dv_dt_index + NODE_BUFFER_SIZE) % NODE_BUFFER_SIZE
```

然后以 `fault_idx` 为中心，前后各取 2 周期（共 40 点）保存为故障快照。

### 4.3 故障响应三级联动

```
正常(NORMAL) ──故障检测──→ 预警(WARNING) ──持续恶化──→ 紧急(DANGER)
    │                          │                        │
    │                          ├ 12kHz 录波 250ms       ├ 15kHz 录波 450ms
    │                          ├ LED1 亮                ├ LED2 闪烁
    │                          └ 5s 健康恢复 → 切回     ├ 备电切换
    │                                                    └ 10s 健康恢复 → 切回
    │
    └ 健康度 > 阈值 且持续 → 恢复正常
```

## 5. 任务详细说明

当前共 **6 个任务**：

### 5.1 data_collect_task（P7, 栈 256 words）

```
while(1) {
    1. 获取当前活跃节点号
    2. 轮转时间索引（0~59），从仿真数组读取数据点
       - 采样计数 30000~30019 → g_zdata_fault[0..19] (模拟故障批次1)
       - 采样计数 50000~50019 → g_zdata_fault[0..19] (模拟故障批次2)
       - 其余时间 → g_zdata_normal[轮转索引]
    3. 拷贝到 SensorData_t → xQueueSend(g_sampledata_queue)
    4. 每 200ms 调用 battery_baseline_collect() 采集电池电压基准
    5. vTaskDelayUntil(1ms)  // 1kHz 节拍
}
```

**数据源**：`zdata_adaptive` 模块中 `g_zdata_normal[60]` + `g_zdata_fault[20]`，均为预定义的 float 仿真数据。
**电池采样**：每 200 次循环（200ms）采集一次电池电压，存入 20 点基准数组，存满后自动计算平均基准值。

### 5.2 data_judge_task（P5, 栈 256 words）

```
while(1) {
    xQueueReceive(g_sampledata_queue, &sensor, portMAX_DELAY)
        → node_sample_process(&sensor.zdata, sensor.timestamp)
}
```

**`node_sample_process()` 内部逻辑**（详见第 4 节）：
- float → int16_t ×10000 定点转换写入环缓冲（40 点环形）
- 每 10 点一次滑动窗口故障检测
- 检测到故障 → `trigger_fault()`：
  - `fault_snap_begin()` 开始故障快照
  - `fault_snap_write_window()` 逐窗口（10 点/窗）写入 Flash
  - `dma_wave_start()` 启动 DAC→ADC→DMA 录波
  - `switch_to_warning_mode()` 或 `switch_to_danger_mode()`
- 故障恢复 → `fault_snap_finalize()` + `switch_to_normal_mode()`

### 5.3 lora_send_task（P6, 栈 512 words）

```
while(1) {
    xQueueReceive(g_lora_send_queue, &packet, portMAX_DELAY)
    1. 组装帧: [0xAA 0x55][len(2B)][timestamp(4B)][type(1B)][sync_code(8B)][payload][CRC8][0x55 0xAA]
    2. send_node_data_with_ack(data, len, data_type, dest, retries=3, timestamp)
       → LoRa_SendData() AT指令发送
    3. 记录发送结果日志
}
```

**两种入队方式**：

| 函数 | 数据大小 | 阻塞/非阻塞 | 用途 |
|------|----------|-------------|------|
| `send_normal_data()` | ≤ 40B | 非阻塞 | 上传 Header（NodeUploadHeader / WaveChunkHeader） |
| `send_waveform_packet()` | ≤ 220B | 阻塞 | 上传原始数据包（节点数据 / 波形分片） |

### 5.4 lora_recv_task（P4, 栈 512 words）

```
while(1) {
    1. 从 LoRa 模块接收数据帧
    2. frame_parse() 提取: type / sync_code / enc_len / payload
    3. 构建 Command_t: cmd_id = rx_type
    4. switch(cmd_id):
         CMD_REQUEST_WAVEFORM(0x10): param[1]=node_idx, param[2]=fault_idx
             → wave_retrieve_by_node_fault(node_idx, fault_idx)
         CMD_POLL_STATUS(0x14): param[1..4] → 32位 poll_ts
             → node_upload_by_timestamp(poll_ts)
         CMD_CLEAR_FLASH(0x12): → wave_flash_init()
}
```

### 5.5 warning_task（P2, 栈 128 words）

等待 `EVENT_WARNING_BIT` 事件：
- LED1 亮，重置恢复计时
- 持续轮询：健康度 > 阈值 且持续 5s → 切回 NORMAL
- 若收到 `EVENT_DANGER_BIT` → 升级到 DANGER

### 5.6 danger_task（P2, 栈 128 words）

等待 `EVENT_DANGER_BIT` 事件：
- LED2 闪烁（100ms 周期）
- 持续轮询：健康度 > 阈值 且持续 10s → 切回 NORMAL
- 健康度 < 30% → 切换备用电源
- 恢复正常后自动 restore_main_power()

## 6. 主控通信协议

### 6.1 主控命令

| 命令码 | 名称 | 参数 | 功能 |
|--------|------|------|------|
| `0x10` | CMD_REQUEST_WAVEFORM | param[1]=节点号, param[2]=故障序号(0-based) | 请求上传指定节点的指定故障波形 |
| `0x14` | CMD_POLL_STATUS | param[1..4]=32位 poll_ts（大端） | 主控轮询，设备按时间戳回溯上传 |
| `0x12` | CMD_CLEAR_FLASH | 无 | 擦除 64KB 波形 Flash 区 |

### 6.2 LoRa 空中帧格式

```
┌──────┬──────┬──────┬──────────┬──────┬──────────┬─────────┬──────┬──────┐
│ 0xAA │ 0x55 │ len  │ timestamp│ type │ sync_code│ payload │ CRC8 │ 0x55 │
│      │      │ 2B(BE)│ 4B(BE)  │ 1B   │ 8B       │  nB     │ 1B   │ 0xAA │
└──────┴──────┴──────┴──────────┴──────┴──────────┴─────────┴──────┴──────┘
```

### 6.3 上传数据类型

| data_type | 名称 | 承载内容 |
|-----------|------|---------|
| `0x01` NODE_HEAD | 节点状态头 | `NodeUploadHeader_t`: severity/health_score/battery_pct/total_points 等 |
| `0x02` WAVE | 波形头 | `WaveChunkHeader_t`: node_index/severity/fault_idx/fault_timestamp/sample_rate/count |
| `0x03` POWER | 电源电压 | 预留 |
| `0x04` NODE_RAW | 节点原始数据 | `NodeSample_t` 数组 |
| `0x05` FLASH_WAVE | 波形原始数据 | 差分编码的 int16 采样点 |
| `0x06` FLASH_WAVE_END | 波形结束标志 | 含总采样点数 |
| `0x07` FAULT_HEAD | 故障快照头 | `NodeUploadHeader_t`（fault_type 非 NONE） |

### 6.4 节点头字段说明

`NodeUploadHeader_t` 各字段含义：

| 字段 | 类型 | 说明 |
|------|------|------|
| data_type | uint8 | `0x01` = 正常轮询头, `0x07` = 故障快照头 |
| severity | uint8 | 故障级别：0=正常, 1=预警, 2=紧急 |
| fault_type | uint8 | 故障类型：0=无, 1=过压, 2=欠压, 3=暂降, 4=暂升 |
| fault_pending | uint8 | 是否有待上传的故障快照 |
| node_index | uint8 | 当前活动节点号 |
| timestamp | uint32 | 时间戳 (ms) |
| sample_rate | uint16 | 采样率 (1000Hz) |
| total_points | uint16 | 后续 raw 数据总点数（正常 20 / 故障 40） |
| battery_pct | uint8 | 电池电量百分比 (0~100)，255=基准值未就绪 |
| health_score | float | 综合健康度评分 |

## 7. 主控时间戳轮询深度解析

这是本系统的**核心交互机制**，替代了传统的定时推送模式。

### 7.1 请求格式

主控发送 `CMD_POLL_STATUS(0x14)`，附带 4 字节大端时间戳 `poll_ts`。

### 7.2 设备端检索（node_upload_by_timestamp）

```
步骤1: SRAM 环缓冲逆向查找
  从 write_index-1 向前遍历 g_node.buffer[40]：
    if (ts <= poll_ts) → 找到 best_idx，break
  遍历方向: 新→旧，保证找到的是"不晚于 poll_ts 的最新数据"

步骤2: 发送 Header
  NodeUploadHeader_t: data_type=0x01, severity=当前级别,
                      battery_pct=当前电池百分比, health_score,
                      total_points=20, fault_type=FAULT_NONE

步骤3: 上传 20 点数据（前 1 周期）
  每次取 4 点打包，发送 NODE_RAW 数据包 (data_type=0x04)
  从 best_idx 往回循环读取

步骤4: 附带上传故障快照
  upload_fault_snaps() → 遍历 PendingFault_t[]，上传所有待发送快照
```

### 7.3 数据检索示例

```
假设 poll_ts = 1200ms，环缓冲中:
  [1150, 1151, 1152, ..., 1198, 1199, 1200, 1201, 1202, ...]
                          ↑   找到 ts=1200 ≤ poll_ts
                          best_idx 指向此处
上传: best_idx 前 20 点 = 20 点（1 周期）
```

## 8. Flash 存储机制

### 8.1 故障快照

- **触发**：`fault_snap_begin()` 初始化，`fault_snap_write_window()` 逐窗口增量写入
- **存储位置**：`NODE_FAULT_SNAP_FLASH_ADDR = 0x08030000`（40KB）
- **每条大小**：12B Header + 40 × 52B Raw = 3132B
- **最多**：5 条（与 MAX_PENDING_FAULTS 一致）
- **写入方式**：逐窗口（10 点/窗）增量写入 Flash，故障结束时 `fault_snap_finalize()` 写入 Header
- **上传后**：`g_pending_fault_count = 0`，下次新故障可覆盖

### 8.2 波形录波数据

- **触发**：`dma_wave_start(rate)`，由 `trigger_fault()` 调用
- **存储位置**：`WAVE_FLASH_ADDR = 0x08020000`（64KB 环形写）
- **DMA 乒乓缓冲**：`dma_buf[2][64]`，半满中断写 Flash。64 点采集完触发中断（12kHz 时 ≈ 5.33ms，15kHz 时 ≈ 4.27ms）
- **Flash 写入**：整页擦除 → FMC 字编程（4B/次）
- **采样率**：预警 12kHz × 250ms = 3000 点，紧急 15kHz × 450ms = 6750 点

## 9. 电池电压监测

### 9.1 硬件配置

| 参数 | 值 | 说明 |
|------|-----|------|
| ADC 通道 | ADC_CHANNEL_10 | PC0 引脚 |
| 参考电压 | 3300 mV | 3.3V VREF |
| ADC 分辨率 | 4095 (12-bit) | |
| 分压比 | 5:1 | 外部分压电阻 |
| ADC 时钟 | 12 MHz | AHB/3 |
| 采样时间 | 239.5 周期 | 约 20μs |
| 触发方式 | 软件触发 (SWRCST) | 单次转换，非连续模式 |

### 9.2 基准值标定流程

```
系统启动
  │
  ├── main.c: battery_monitor_init()    // 初始化 ADC_CH10、PC0
  ├── main.c: battery_baseline_init()   // 清零基准数组
  │
  └── data_collect_task 每 200ms:
        └── battery_baseline_collect()
              ├── 调用 battery_monitor_sample() 读取 ADC
              ├── 存入 g_battery_baseline_arr[count]
              ├── count++
              └── count == 20 时:
                    ├── 计算平均值 → g_battery_baseline_mv
                    └── 停止采集（后续调用直接返回）
```

20 点 × 200ms = **4 秒**完成基准值标定。

### 9.3 百分比计算与上传

```
每次发送节点头时 (node_upload_by_timestamp / upload_fault_snaps):
  │
  └── battery_get_percentage()
        ├── 若基准值未就绪 → 返回 0xFF
        ├── 调用 battery_monitor_sample() 获取当前电压
        └── return (当前电压 × 100 / 基准电压)，上限 100
```

### 9.4 关键函数

| 函数 | 功能 |
|------|------|
| `battery_monitor_init()` | 初始化 ADC_CH10、PC0 模拟输入 |
| `battery_monitor_sample()` | 软件触发一次 ADC 转换，更新 `g_battery_mv` |
| `battery_get_voltage_mv()` | 返回最近一次采样的电压值 (mV) |
| `battery_baseline_init()` | 清零基准数组，重置计数器 |
| `battery_baseline_collect()` | 采样一次并存入基准数组，满 20 点后计算平均值 |
| `battery_baseline_ready()` | 返回基准值是否已就绪 (1/0) |
| `battery_get_baseline_mv()` | 返回基准电压平均值 (mV) |
| `battery_get_percentage()` | 返回当前电压占基准值的百分比 (0~100)，255=未就绪 |

### 9.5 RAM 开销

| 变量 | 大小 |
|------|------|
| `g_battery_mv` | 2B |
| `g_battery_baseline_arr[20]` | 40B |
| `g_battery_baseline_count` | 1B |
| `g_battery_baseline_mv` | 2B |
| **合计** | **45B** |
