> **更新**: 2026-06-05 | **状态**: 三节点并行已验证, 100% 成功率

## 一、当前做了什么

### 1.1 三节点 FreeRTOS 并行仿真

在 FreeRTOS 侧创建了三个独立任务, 同时运行不同规模的电力系统动态仿真:

| 任务 | 系统 | 发电机 | 母线 | 状态维 | 测量维 | 步长 |
|------|------|:--:|:--:|:--:|:--:|------|
| SIM | IEEE 5-Bus | 2 | 5 | 4 | 14 | 0.5ms |
| SIM39 | IEEE 39-Bus | 10 | 39 | 20 | 98 | 0.5ms |
| SIM9 | IEEE 9-Bus | 3 | 9 | 6 | 24 | 0.5ms |

每个任务独立运行 RK4 数值积分, 产生 360,000 帧(180s 仿真时长 @ 2000Hz)。

### 1.2 数据通信架构

```
FreeRTOS (CPU1)                    Linux (CPU0/2/3)
═══════════════                    ════════════════
SIM (5bus) ─┐                     bridge_three
SIM39       ├─ RPMsg ──────────→    ├─ fd=3 (5bus)
SIM9        ─┘  统一通道            ├─ fd=4 (39bus)
            rpmsg-sim-ctrl          └─ fd=5 (9bus)
                                     ↓
                                  SHM / Pipe / UKF
```

### 1.3 RPMsg 统一通道方案

三个 SIM 任务使用**同一个 RPMsg 通道名** `rpmsg-sim-ctrl`, 但分配不同 src 地址:

| 任务 | src 地址 | START 命令码 |
|------|:------:|:----------:|
| SIM (5bus) | 0x0 | 0x51 |
| SIM39 | 0xa (10) | 0x60 |
| SIM9 | 0x14 (20) | 0x70 |

Linux 端 `bridge_three` 程序创建三个独立 RPMsg endpoint, 各自 dst 精确匹配 FreeRTOS src 地址, 分别发送 START。

**为什么这样做**: OpenAMP 的 `rpmsg_create_ept` 在创建多个 endpoint 时会自动分配地址, 可能导致两个任务分配到相同地址。使用统一通道+显式指定不同 src 地址解决了这个问题。

### 1.4 关键修复: RPMsg dest_addr 更新

**问题**: 桥接重启后 FreeRTOS 仍向旧 Linux 端点地址发送数据, 导致 `rpmsg_send` 虽然返回成功但数据丢失。

**修复**: 在每个 SIM 任务的 RPMsg 回调中, 收到 START 命令时显式更新 `ept->dest_addr = src`:

```c
if (ctrl->cmd == SIM_CTRL_START) {
    if (src != RPMSG_ADDR_ANY && ept) ept->dest_addr = src;
    sim_running = 1;
    ...
}
```

### 1.5 共享内存数据传输

为避免 RPMsg 接收路径的潜在问题, 同时实现了共享内存(SHM)直通方案:

- FreeRTOS 每帧测量数据直接写入 `0xC8100000-0xC811FFFF` 的环形缓冲区
- Linux 端 `shm_reader` 通过 `/dev/mem` 读取, 无需 RPMsg 接收路径
- 支持三个节点各自 32KB~64KB 缓冲区

## 二、怎么做的

### 2.1 FreeRTOS 任务结构

每个 SIM 任务 (`sim_node_task.c`, `sim_node_39bus.c`, `sim_node_9bus.c`) 包含:

1. 系统参数(硬编码为 C 头文件, Python 脚本自动生成)
2. RK4 积分函数(使用 `float` 单精度)
3. 测量函数 `h(x)` (状态→测量向量)
4. RPMsg endpoint 创建 + START 等待
5. 主循环: RK4→测量→RPMsg 发送/SHM 写入
6. 故障切换: 5.0s~5.3s 自动切换 YBUS/RV 矩阵

### 2.2 Linux 端桥接

`bridge_three.c`: 创建 3 个 RPMsg endpoint, 分别向 3 个 SIM 任务发送 START:

```c
int f5  = open_channel("rpmsg-sim-ctrl", 0x0);   // SIM5 src=0
int f39 = open_channel("rpmsg-sim-ctrl", 0xa);   // SIM39 src=10
int f9  = open_channel("rpmsg-sim-ctrl", 0x14);  // SIM9 src=20
send_cmd(f5, 0x51);  // START 5bus
send_cmd(f39, 0x60); // START 39bus
send_cmd(f9, 0x70);  // START 9bus
```

### 2.3 性能指标(Phytium ARM64)

| 指标 | 5bus/2-gen | 39bus/10-gen | 9bus/3-gen |
|------|:--------:|:----------:|:--------:|
| RK4 每步耗时 | ~6.5μs | ~50μs | ~15μs |
| 全部 360k 帧耗时(全速) | ~2s | ~18s | ~5s |
| RPMsg 成功率 | 100% | 100% | 100% |
| C UKF 处理速度 | 1,804 fps | ~300 fps(估) | ~800 fps(估) |

## 三、踩坑记录

### 坑1: RPMsg dest_addr 必须每次更新

上面 1.4 已详述。核心教训: 不要依赖 OpenAMP 的首次自动绑定, 每次收到消息都更新 dest_addr。

### 坑2: OpenAMP 端点地址冲突

创建 3+ 个 endpoint 时, src 地址可能重复。解决: 显式指定不同的 src 地址(0, 10, 20)。

### 坑3: SIM 任务在桥接连接前就跑完

SIM 任务默认 speed=1(实时), 180s 仿真需 180s。但如果 speed=0, 几秒跑完。需要桥接在 FreeRTOS 启动后尽快连接。

### 坑4: 测量函数格式不统一

不同系统使用不同的电压测量格式:
- 5bus/39bus: 幅值+相角 (`|V|` + `∠V`)
- 9bus: 实部+虚部 (`Re(V)` + `Im(V)`)

UKF 需要对应适配。

### 坑5: SHM 地址与 trace_reader 冲突

初始 SHM 地址 `0xC8001000` 落在 trace_reader 的环形缓冲区内, 数据被覆盖。解决: SHM 移到 `0xC8100000`, 扩展 MMU 映射到 2MB。
