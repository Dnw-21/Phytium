# GD32L233C 边缘自治电网监测系统

## 1. 系统概述

本项目基于 GD32L233C 微控制器（ARM Cortex-M23, **32KB RAM**, **256KB Flash**）实现电网三相电压/功率信号的采集、故障检测、数据分级存储及主控按需拉取上传的完整自治闭环系统。采用 FreeRTOS 多任务调度，通过 LoRa 无线模块（ATK-MWCC68D）与主控通信。

### 1.1 核心亮点

| 亮点 | 说明 |
|------|------|
| **主控时间戳轮询** | 不再定时推送，改为主控下发 `poll_ts` 时间戳 → 设备回溯查找对应时刻前后 2 周期数据上传。**按需拉取，零冗余传输**。 |
| **SRAM + Flash 双级数据回溯** | 200 点 SRAM 环形缓冲（最新 10 周期） + 40KB Flash 批次归档（含时间索引表）。`poll_ts` 先在 SRAM 查找，未命中则从 Flash 回退检索。 |
| **int16 定点数压缩存储** | `NodeSample_t` 全部字段 ×10000 转为 `int16_t`，每采样点从 36B → **20B**（节省 44%）。8 个 int16 两两合并为 4 个 int32，上传一包最多承载 12 点。 |
| **三相通用字段命名** | `voltage_mag_a/b/c`、`voltage_angle_a/b/c`，不带通道号，适配不同节点（不同节点可能映射不同物理通道）。 |
| **3 级 Flash 分级存储** | 节点数据（40KB 环写 + 索引表） | 故障快照（4KB，最多 5 条） | 波形原始数据（64KB 环写） |
| **10ms 滑动窗口故障检测** | 每 10 个采样点触发一次 RMS / 最大差分斜率(dv_dt) / 峰峰值(peak) 三特征综合判断，精准定位故障点位置（精确到窗口内采样点索引）。 |
| **DAC → 导线 → ADC 硬件在环录波** | DAC 输出正弦波经外接导线 → ADC 采集 → DMA 乒乓缓冲 → Flash 存储。预警 12kHz/250ms，紧急 15kHz/450ms。 |
| **波形差分编码压缩上传** | 相邻采样点差值 <128 时用 1B 编码，否则用 3B（标志+原始），显著压缩传输体积。 |

### 1.2 主要功能

- **三相电力数据采集**：1kHz 采样，50Hz 每周期 20 点，10 周期（200 点）SRAM 环形缓冲
- **中频故障检测**：10ms / 10 点滑动窗口，实时检测电压暂升/暂降/过压/欠压
- **分级故障响应**：正常(NORMAL) → 预警(WARNING, 12kHz 录波 250ms) → 紧急(DANGER, 15kHz 录波 450ms + 备用电源切换)
- **3 级 Flash 存储**：节点数据归档(40KB) + 故障快照(4KB) + 录波数据(64KB)
- **命令驱动上传**：主控下发请求 → 设备按需上传，支持时间戳轮询、故障波形检索、Flash 擦除

### 1.3 硬件资源分配

| 外设 | 用途 | 说明 |
|------|------|------|
| TIMER2 | 波形采样定时器 | 触发 DAC/ADC，频率 12kHz / 15kHz |
| DAC (PA4) | 正弦波输出 | DMA_CH2 循环，256 点 LUT |
| ADC (PA0) | 波形采集 | DMA_CH1，乒乓缓冲（2×64 点） |
| DMA_CH1 | ADC 数据传输 | 半满/满中断触发 Flash 写入 |
| DMA_CH2 | DAC 数据传输 | 循环模式，自动输出正弦波 |
| USART1 (PA2/PA3) | LoRa 通信 | 115200bps，ATK-MWCC68D，混沌加解密 |
| USART0 (PA9/PA10) | 调试日志 | 115200bps，printf 重定向 |

### 1.4 Flash 存储布局

| 区域 | 起始地址 | 大小 | 用途 | 写入粒度 |
|------|----------|------|------|----------|
| 程序代码 | 0x08000000 | ~128KB | 固件 | - |
| 波形数据 | 0x08020000 | **64KB** | DMA 录波原始数据（环形覆盖） | 64 点/次 |
| 节点数据 | 0x08030000 | **40KB** | 环缓冲批量落盘 + 10 条时间索引 | 4096B 页 |
| 故障快照 | 0x0803A000 | **4KB** | 故障时刻前后各 2 周期快照（最多 5 条） | 812B/条 |

## 3. 数据结构（重点）

### 3.1 节点采样点 — NodeSample_t（int16 定点数压缩）

```c
typedef struct {
    int16_t active_power;        // 有功功率 ×10000
    int16_t reactive_power;      // 无功功率 ×10000
    int16_t voltage_mag_a;       // A相电压幅值 ×10000
    int16_t voltage_mag_b;       // B相电压幅值 ×10000
    int16_t voltage_mag_c;       // C相电压幅值 ×10000
    int16_t voltage_angle_a;     // A相电压相角 ×10000
    int16_t voltage_angle_b;     // B相电压相角 ×10000
    int16_t voltage_angle_c;     // C相电压相角 ×10000
    uint32_t timestamp;          // 时间戳 ms
} NodeSample_t;  // 每点 8×2 + 4 = 20B
```

> **设计意图**：三相电力监测的标准数据模型——每个节点包含 1 组有功/无功、3 路电压幅值、3 路相角。使用 `int16_t ×10000` 压缩存储，牺牲少量精度（0.1mV/0.1m°）换取 44% 的 RAM 节省。

### 3.2 环缓冲 — NodeBuffer_t

```c
typedef struct {
    NodeSample_t buffer[200];    // 200 点 = 10 周期 = 4000B
    uint16_t write_index;        // 当前写入位置 (环形)
    uint16_t cycle_count;        // 累计采样点计数（溢出后回绕，仅用于判断是否满 2 周期）
} NodeBuffer_t;
```

> `cycle_count` 达到 40（2 周期）后才启动故障检测；达到 200（10 周期满）后每次累积整批时会自动落盘到 Flash。

### 3.3 Flash 批次索引 — NodeFlashIdx_t

```c
typedef struct {
    uint32_t flash_off;          // Flash 中该批次的起始偏移
    uint32_t start_ts;           // 该批次第一个采样点的时间戳
} NodeFlashIdx_t;
```

最多 10 条索引（`NODE_FLASH_MAX_INDEX = 10`），覆盖 10×4096B = 40KB。主控时间戳轮询时，若 SRAM 环缓冲未命中（数据已落盘），则遍历此索引表找到对应批次。

### 3.4 故障快照记录 — PendingFault_t

```c
typedef struct {
    uint32_t fault_time;         // 故障发生的时间戳
    uint32_t flash_off;          // 快照在 NODE_FAULT_SNAP_ADDR 中的偏移
    uint8_t  severity;           // 故障级别 (SEVERITY_WARNING / DANGER)
    uint8_t  fault_type;         // 故障类型
    uint8_t  valid;              // 1=待上传
} PendingFault_t;
```

每条快照占 Flash 812B = 12B Header(fault_time + severity + fault_type + padding) + 40 × 20B 原始 `NodeSample_t`。最多 5 条（MAX_PENDING_FAULTS=5），在每次主控轮询后附带上传。

### 3.5 中频滑动窗口 — MidFreqWindow_t

```c
typedef struct {
    float voltage_mag[10];       // A相电压幅值滑动窗口（10 点 = 10ms）
    uint16_t index;              // 窗口写入索引
    float rms;                   // 窗口 RMS（最近一次计算）
    float dv_dt;                 // 最大相邻差分斜率（最近一次计算）
    uint16_t dv_dt_index;        // 最大差分斜率发生的窗口内索引
    float peak;                  // 窗口峰峰值（max - min）
    float thd;                   // THD 总谐波失真（预留）
} MidFreqWindow_t;
```

> 从 20 点窗口精简为 10 点（与 NODE_BUFFER_SIZE=200、SAMPLES_PER_CYCLE=20 的比例一致）。当前仅使用 A 相电压幅值进行故障检测。

### 3.6 故障记录 — FaultRecord_t

```c
typedef struct {
    uint32_t timestamp;          // 故障检测时间戳
    FaultType_t fault_type;      // 故障类型（暂升/暂降/过压/欠压）
    SeverityLevel_t severity;    // 故障级别（警告/危险）
    float fault_value;           // 故障时刻 RMS 值
    uint32_t flash_offset;       // 对应波形在 Flash 中的起始偏移
    uint32_t sample_count;       // 波形采样点数
    uint16_t sample_rate;        // 波形采样率（12kHz/15kHz）
    uint8_t  valid;              // 有效标志（上传后置 0）
    uint8_t  node_index;         // 故障节点号
} FaultRecord_t;
```

最多 5 条（`MAX_FAULT_RECORDS=5`），上传后自动标记无效并压缩数组。

### 3.7 传感器数据 — SensorData_t（任务间传递）

```c
typedef struct {
    uint32_t timestamp;
    float active_power;
    float reactive_power;
    float voltage_mag_a;
    float voltage_mag_b;
    float voltage_mag_c;
    float voltage_angle_a;
    float voltage_angle_b;
    float voltage_angle_c;
} SensorData_t;  // 36B
```

> 这是 `data_collect_task` → `g_sampledata_queue` → `data_judge_task` 的传递载体，用 `float` 保留原始精度供故障检测使用。进入 `node_sample_process()` 后转为 `int16_t` 定点存入环缓冲。

## 4. 故障检测机制

### 4.1 检测流程

```
node_sample_process() 每 1ms 被调用一次
    │
    ├── 写入环缓冲 + cycle_count++
    │
    ├── 每 200 点整批 → Flash 批量落盘（4096B 页对齐）
    │
    ├── cycle_count < 40 时跳过检测（不足 2 周期）
    │
    └── 每累计 10 个采样点 触发一次故障检测:
            │
            ├── 将最新 10 个 A 相电压幅值点加载到 MidFreqWindow_t.voltage_mag[10]
            │
            ├── detect_mid_freq_fault() 计算:
            │   ├── RMS = sqrt(Σv² / 10)
            │   ├── Peak = max - min
            │   └── dv_dt = max(|v[i+1] - v[i]|), 记录 dv_dt_index
            │
            ├── health_calculate(rms, dv_dt, peak) → 更新综合健康度
            │
            └── 电压阈值判断:
                    RMS > 1.10pu → FAULT_VOLTAGE_SWELL  (暂升)
                    RMS > 1.05pu → FAULT_OVER_VOLTAGE    (过压)
                    RMS < 0.90pu → FAULT_VOLTAGE_SAG     (暂降)
                    RMS < 0.95pu → FAULT_UNDER_VOLTAGE   (欠压)
```

### 4.2 dv_dt 故障点精确定位

`dv_dt_index` 记录窗口内最大相邻差分斜率的位置。故障发生时，利用此索引反算出环缓冲中故障发生的精确位置：

```
fault_idx = (window_pos - 10 + dv_dt_index + NODE_BUFFER_SIZE) % NODE_BUFFER_SIZE
```

然后以 `fault_idx` 为中心，前后各取 2 周期（共 40 点）保存为故障快照。

### 4.3 故障响应三级联动

```
正常(NORMAL) ──故障检测──→ 预警(WARNING) ──持续恶化──→ 紧急(DANGER)
    │                          │                        │
    │                          ├ 12kHz 录波 250ms       ├ 15kHz 录波 450ms
    │                          ├ LED1 亮                ├ LED2 闪烁
    │                          └ 5s 健康恢复 → 切回     ├ 备电切换
    │                                                    └ 10s 健康恢复 → 切回
    │
    └ 健康度 > 阈值 且持续 → 恢复正常
```

## 5. 任务详细说明

当前共 **6 个任务**（精简自原来的 8 个，去掉了 `status_upload_task` 和 `wave_upload_task`）：

### 5.1 data_collect_task（P7, 栈 256 words）

```
while(1) {
    1. 获取当前活跃节点号
    2. 生成时间戳（tick × portTICK_PERIOD_MS）
    3. 轮转时间索引（0~59），从仿真数组读取数据点
       - 前 50000 次 → g_node_normal[轮转索引]
       - 接下来 20 次 → g_node_fault[顺序索引]（模拟故障）
       - 之后继续 → g_node_normal[]
    4. 拷贝到 SensorData_t → xQueueSend(g_sampledata_queue)
    5. vTaskDelayUntil(1ms)
}
```

**数据源**：`node1_data.c` 中 `g_node_normal[60]` + `g_node_fault[20]`，均为预定义的仿真数据（可在后续新增 node2_data.c、node3_data.c...）。

### 5.2 data_judge_task（P5, 栈 256 words）

```
while(1) {
    xQueueReceive(g_sampledata_queue, &sensor, 50ms)
        → node_sample_process(active_p, reactive_p,
                              vmag_a, vmag_b, vmag_c,
                              vang_a, vang_b, vang_c,
                              timestamp)
}
```

**`node_sample_process()` 内部逻辑**（详见第 4 节）：
- float → int16_t ×10000 定点转换写入环缓冲
- 每 200 点整批落盘到 Flash（4000B → 4096B 页）
- 每 10 点一次滑动窗口故障检测
- 检测到故障 → `trigger_fault()`：
  - `node_fault_save()` 保存 40 点快照到 0x0803A000
  - `dma_wave_start()` 启动 DAC→ADC→DMA 录波
  - `switch_to_warning_mode()` 或 `switch_to_danger_mode()`
- 故障恢复 → `switch_to_normal_mode()`

### 5.3 lora_send_task（P6, 栈 512 words）

```
while(1) {
    xQueueReceive(g_lora_send_queue, &packet, portMAX_DELAY)
    1. 数据拷贝到 g_encrypted_buffer[128]
    2. 组装帧: [sync_code(4B)][payload]（当前明文传输，加密被注释）
    3. send_node_data_with_ack(data, len, data_type, dest, retries=3, timestamp)
       → 帧格式: [0xAA 0x55][len(2B)][timestamp(4B)][type(1B)][payload][CRC8][0x55 0xAA]
       → LoRa_SendData() AT指令发送
    4. 记录发送结果日志
}
```

**两种入队方式**：

| 函数 | 数据大小 | 阻塞/非阻塞 | 用途 |
|------|----------|-------------|------|
| `send_normal_data()` | ≤ 40B | 非阻塞 | 上传 Header（NodeUploadHeader / WaveChunkHeader） |
| `send_waveform_packet()` | ≤ 220B | 阻塞 | 上传原始数据包（节点数据 / 波形分片） |

### 5.4 lora_recv_task（P4, 栈 512 words）

```
while(1) {
    1. rx_len = sizeof(rx_buf)
    2. atk_mwcc68d_receive_data(rx_buf, &rx_len, 100ms)
    3. 软超时检测: rx_count 连续 20ms 不变 → 一帧完整
    4. frame_parse() 提取: type / sync_code / enc_len / payload
    5. chaos_decrypt_packet() 解密（当前明文跳过）
    6. 构建 Command_t: cmd_id = rx_type, param = payload[0..15]
    7. switch(cmd_id):
         CMD_REQUEST_WAVEFORM(0x10): param[1]=node_idx, param[2]=fault_idx
             → wave_retrieve_by_node_fault(node_idx, fault_idx)
         CMD_POLL_STATUS(0x14): param[1..4] → 32位 poll_ts
             → node_upload_by_timestamp(poll_ts)
         CMD_CLEAR_FLASH(0x12): → wave_flash_init()
}
```

### 5.5 warning_task（P2, 栈 128 words）

等待 `EVENT_WARNING_BIT` 事件：
- LED1 亮，重置恢复计时
- 持续轮询：健康度 > 阈值 且持续 5s → 切回 NORMAL
- 若收到 `EVENT_DANGER_BIT` → 升级到 DANGER

### 5.6 danger_task（P2, 栈 128 words）

等待 `EVENT_DANGER_BIT` 事件：
- LED2 闪烁（100ms 周期）
- 持续轮询：健康度 > 阈值 且持续 10s → 切回 NORMAL
- 健康度 < 30% → 切换备用电源（GPIO PC0 控制继电器）
- 恢复正常后自动 restore_main_power()

## 6. 主控通信协议

### 6.1 主控命令

| 命令码 | 名称 | 参数 | 功能 |
|--------|------|------|------|
| `0x10` | CMD_REQUEST_WAVEFORM | param[1]=节点号, param[2]=故障序号(0-based) | 请求上传指定节点的指定故障波形 |
| `0x14` | CMD_POLL_STATUS | param[1..4]=32位 poll_ts（大端） | 主控轮询，设备按时间戳回溯上传 |
| `0x12` | CMD_CLEAR_FLASH | 无 | 擦除 64KB 波形 Flash 区 |

### 6.2 LoRa 空中帧格式

```
┌──────┬──────┬──────┬──────────┬──────┬──────────┬─────────┬──────┬──────┐
│ 0xAA │ 0x55 │ len  │ timestamp│ type │ sync_code│ payload │ CRC8 │ 0x55 │
│      │      │ 2B(BE)│ 4B(BE)  │ 1B   │ (明文=0) │  nB     │ 1B   │ 0xAA │
└──────┴──────┴──────┴──────────┴──────┴──────────┴─────────┴──────┴──────┘
```

### 6.3 上传数据类型

| data_type | 名称 | 承载内容 |
|-----------|------|---------|
| `0x01` STATUS | 节点头 | `NodeUploadHeader_t`: data_type/severity/health_score/total_points/fault_type |
| `0x04` NODE_RAW | 节点原始数据 | 10 点 NodeSample 合并打包：8×int16 → 4×int32[10] + timestamp[10] |
| `0x02` WAVE | 波形头 | `WaveChunkHeader_t`: node_index/severity/fault_idx/fault_timestamp/sample_rate/count |
| `0x05` FLASH_WAVE | 波形原始数据 | 差分编码的 int16 采样点 |

### 6.4 上传数据压缩 — int16 合并打包

每个采样点 8 个 `int16_t` 两两合并为 4 个 `int32_t`，加 1 个 `timestamp`（int32）：

```
包内排列 (10 点为例):
  pkt[0..3]   = [reactive_power|active_power], [vmag_b|vmag_a], [vang_a|vmag_c], [vang_c|vang_b]
  pkt[4]      = timestamp_0
  pkt[5..8]   = 点1 ...
  ...
```

每包 240B 可承载 **(240-4) / (4×4+4) ≈ 11 个点**，实际上传时每次发 **10 点**（保留余量）。

### 6.5 波形差分编码压缩

上传 Flash 中的录波原始数据时，对相邻 int16 采样点做差分编码：

```
delta = sample[i] - sample[i-1]
if (-128 < delta < 127):
  输出 1B (int8_t)delta
else:
  输出 3B: [0x80] [sample 高8位] [sample 低8位]
```

波形数据通常变化平缓，差分值大多落在 1B 范围内，**平均压缩比约 40%~60%**。

## 7. 主控时间戳轮询深度解析

这是本系统的**核心交互机制**，替代了传统的定时推送模式。

### 7.1 请求格式

主控发送 `CMD_POLL_STATUS(0x14)`，附带 4 字节大端时间戳 `poll_ts`。

### 7.2 设备端检索（node_upload_by_timestamp）

```
步骤1: SRAM 环缓冲逆向查找
  从 write_index-1 向前遍历 g_node.buffer[200]：
    if (ts <= poll_ts) → 找到 best_idx，break
  遍历方向: 新→旧，保证找到的是"不晚于 poll_ts 的最新数据"

步骤2: Flash 回退查找（SRAM 未命中时）
  从最新的 Flash 索引向前遍历 NodeFlashIdx_t[10]：
    if (start_ts <= poll_ts) → 定位批次
    flash_pt_off = min(poll_ts - start_ts, 199)
    → 从 Flash 中读取对应点

步骤3: 发送 Header
  NodeUploadHeader_t: data_type=0x01, severity=当前级别,
                      health_score, total_points=40, fault_type=NONE

步骤4: 上传 40 点数据（前后各 2 周期）
  每次取 10 点打包，发送 NODE_RAW 数据包 (data_type=0x04)
  - SRAM 路径: 从 best_idx 往回循环读取
  - Flash 路径: 从 Flash 批次中顺序读取

步骤5: 附带上传故障快照
  upload_fault_snaps() → 遍历 PendingFault_t[]，上传所有待发送快照
```

### 7.3 数据检索示例

```
假设 poll_ts = 1200ms，环缓冲中:
  [1150, 1151, 1152, ..., 1198, 1199, 1200, 1201, 1202, ...]
                          ↑   找到 ts=1200 ≤ poll_ts
                          best_idx 指向此处
上传: best_idx 前 20 点 + best_idx 后 20 点 = 40 点
```

## 8. Flash 存储机制

### 8.1 节点数据批量落盘

- **触发条件**：`cycle_count` 是 200 的整数倍（即每 10 周期触发一次）
- **落盘流程**：
  1. 擦除目标页（`node_flash_erase_page`）
  2. 分片写入 4000B（200 点 × 20B），每次 500B/chunk
  3. 记录索引 `NodeFlashIdx_t { flash_off, start_ts }`
  4. `flash_off += 4096`（页对齐），循环覆盖 40KB
- **容量计算**：40KB / 4KB = 10 批，每批 200 点 = **共可归档 2000 点**（100 周期 ≈ 2 秒）

### 8.2 故障快照

- **触发**：`trigger_fault()` 内调用 `node_fault_save()`
- **存储位置**：`NODE_FAULT_SNAP_ADDR = 0x0803A000`（4KB）
- **每条大小**：12B Header + 40 × 20B Raw = 812B
- **最多**：5 条（与 MAX_PENDING_FAULTS 一致）
- **上传后**：`g_pending_fault_count = 0`，下次新故障可覆盖

### 8.3 波形录波数据

- **触发**：`dma_wave_start(rate)`
- **存储位置**：`WAVE_FLASH_ADDR = 0x08020000`（64KB 环形写）
- **DMA 乒乓缓冲**：`dma_buf[2][64]`，半满中断写 Flash。64 点采集完触发中断（12kHz 时 ≈ 5.33ms，15kHz 时 ≈ 4.27ms）
- **Flash 写入**：整页擦除 → FMC 字编程（4B/次）
