## Linux 侧方案 — 已验证通过的配置

### 硬件连接

| LoRa 模块 | 飞腾派 J1 | 物理引脚 | 功能 |
|-----------|----------|---------|------|
| VCC | Pin1 | 3.3V | 电源 (3.3V!) |
| GND | Pin6 | GND | 地 |
| RXD | Pin8 | UART2_TXD | LoRa收 ← 飞腾发 |
| TXD | Pin10 | UART2_RXD | LoRa发 → 飞腾收 |
| MD0 | Pin11 | GPIO3_1 | 模式控制 |
| AUX | Pin7 | GPIO2_10 | 忙闲检测 |

### UART 端口: `/dev/ttyAMA2` (不是 ttyAMA3!)

**重要纠正**: J1 Pin8/Pin10 是 **UART2** (`/dev/ttyAMA2`, `0x2800E000`)，不是 UART3。之前所有文档标 UART3 都是错的。

验证通过：对 `/dev/ttyAMA2` 115200-8N1 的 TX/RX 均正常。

### GPIO 控制

- **FIOPAD IOMUX 是必须的** — 这是之前 FreeRTOS 侧 GPIO_DDR 写不进去的根本原因
- GPIO3_1 (MD0): FIOPAD_C49_REG0_OFFSET (0x00E0) → func6
- GPIO2_10 (AUX): FIOPAD_A37_REG0_OFFSET (0x00C4) → func6
- MD0 高低电平切换已验证通过
- AUX 变化已验证（0.7s/3.6s 短暂 HIGH → 收到 RF 数据）

### AT 命令格式

此 LoRa 模块使用 **ATK 原厂格式** (atk_mwcc68d.c 风格)，不是 GD32 BSP 格式：

| 参数 | 正确命令 | GD32 BSP命令 | 结果 |
|------|---------|-------------|------|
| 地址 | `AT+ADDR=00,0B` | `AT+ADDR=00,0B` | ✅ 两者一致 |
| 信道+速率 | `AT+WLRATE=23,5` | `AT+CHN=23` + `AT+WLRATE=5` | ❌ BSP格式返回ERROR |
| 功率 | `AT+TPOWER=4` | `AT+POWER=4` | ❌ BSP格式返回ERROR |
| 波特率 | `AT+UART=7,0` | `AT+BPS=7` | ❌ BSP格式返回ERROR |
| 保存 | `AT+FLASH=1` | 无 | — |

### AUX 排查结论

AUX 引脚完全正常。之前 AUX 一直 LOW 的原因：退出 AT 模式后只等了 1 秒，模块还没完成重启。

**正确做法（照搬 GD32 `LoRa_ExitConfigMode`）**：退出 AT 模式后等待 AUX 变 HIGH，最多等 5 秒。

## GD32 终端节点帧格式

接收到的数据帧：
```
AA 55 [len_hi][len_lo] [dev_id_hi][dev_id_lo] [timestamp...] [payload...]
```

- `AA 55`: GD32 协议帧头
- `len`: 数据长度 (big-endian)
- `dev_id`: 设备ID (0x0009 或 0x000A)
- `timestamp`: 4字节
- `payload`: 加密数据

两种帧类型：
- `00 89` (137 bytes): 来自 dev_id=0x0009 / 0x000A 的长帧
- `00 19` (25 bytes): 短帧

## 后续移植到 FreeRTOS 的注意事项

1. **UART 端口**: 使用 UART2 (0x2800E000)，不是 UART3
2. **GPIO**: 必须先配置 FIOPAD IOMUX 再操作 GPIO 寄存器
3. **Linux 冲突**: UART2 也被 Linux 设备树占用(`status=okay`)，需禁用或处理
4. **AT 格式**: 以模块实际响应的 ATK 格式为准
