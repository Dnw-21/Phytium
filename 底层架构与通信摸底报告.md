### 1.2 设备树 `reserved-memory` (`/proc/device-tree/reserved-memory/`)

| 节点 | reg | 大小 | 含义 |
|------|-----|------|------|
| `rproc@b0100000` | `<0x00 0xb0100000  0x00 0x19900000>` | **409 MiB** | **OpenAMP 共享内存** (`no-map` 标志存在) |

**精确范围**:
- 起始: `0x00000000_b0100000` (2817 MiB)
- 结束: `0x00000000_c9a00000` (3226 MiB)
- 大小: `0x19900000` (419430400 字节 = 409 MiB)
- 属性: `#address-cells=2`, `#size-cells=2`, `no-map` (Linux 不会自动映射, 由 remoteproc 管理)

**落在该保留区内的 SHM 数据区**:
| 节点 | 基地址 | 长度 | 帧大小 |
|------|--------|------|--------|
| 5bus | 0xC8100000 | 32 KiB | 64 B |
| 39bus | 0xC8108000 | 64 KiB | 200 B (实测 400B, 与头文件不一致,见下) |
| 9bus | 0xC8118000 | 32 KiB | 104 B |

→ 所有 SHM 数据都落在 FreeRTOS+OpenAMP 409 MiB 保留区内, Linux 端通过 `mmap /dev/mem` 访问。

### 1.3 物理内存总账 (与开发板硬件一致)

| 区域 | 地址 | 用途 | 大小 |
|------|------|------|------|
| 0x80000000-0xb00fffff | Linux DRAM 区 1 | Kernel+Linux lowmem | 769 MiB |
| 0xb0100000-0xc99fffff | **保留 (rproc)** | **FreeRTOS + OpenAMP SHM** | **409 MiB** |
| 0xc9a00000-0xfaffffff | Linux DRAM 区 2 | Linux highmem+GPU 保留 | 786 MiB |
| 0x2000000000-0x217fffffff | Linux DRAM 区 3 | 4 GiB 以上内存 | ~1.5 GiB |
| **合计** | | | **~3.5 GiB Linux + 409 MiB rproc + 残段 ≈ 8 GiB** |

> 与 `free -h` 输出 `total=7.3 GiB` 一致 (剩余 ~4.5 GiB 可能被 GPU/PCIe 设备占用, 不归 Linux 管)。

### 1.4 `/proc/cpuinfo` 核心拓扑 (重大发现)

```
processor 0 : part=0x303, max=1500 MHz, cur=750 MHz,  BogoMIPS=100   <-- A55 小核
processor 1 : part=0x303, max=1500 MHz, cur=1500 MHz, BogoMIPS=100   <-- A55 小核
processor 2 : part=0x664, max=1800 MHz, cur=900 MHz,  BogoMIPS=100   <-- A76 大核
processor 3 : part=0x664, max=1800 MHz, cur=1800 MHz, BogoMIPS=100   <-- A76 大核
```

**关键点**:
- **物理拓扑不是按 cluster 编号的**: dmesg 显示
  - Linux CPU 0/1 = 物理 cluster 1, A55 (max 1.5 GHz)
  - Linux CPU 2/3 = 物理 cluster 0, A76 (max 1.8 GHz)
- **CPU 0/1 是大核还是小核**: 小核 (A55, 1.5 GHz max)
- **CPU 2/3 才是大核** (A76, 1.8 GHz max)
- **当前 taskset 测试**: `taskset -c 1` 和 `taskset -c 3` **都通过** — CPU3 实际可用
- **CPUs governors**: 全是 `schedutil` (动态调频, 空闲时降频, 见 `scaling_cur_freq`)

### 1.5 不同核使用哪些内存

```
Kernel code/data: 0x80090000-0x819bffff (~25 MiB)        任何核共享

```

```
rootfstype=ext4 rootwait rw cma=256m ;
```

| 参数 | 值 | 作用 |
|------|----|------|
| `console` | `ttyAMA1,115200` | 控制台输出到 UART1 |
| `earlycon` | `pl011,0x2800d000` | 早期串口 (调试) |
| `root` | `/dev/mmcblk0p1` | SD 卡第一个分区 |
| `rootfstype` | `ext4` | 根文件系统 |
| `cma` | `256 MiB` | **CMA (连续内存) 256 MiB** — 视频/网络等需要大块连续内存 |
| `;` | (末尾) | 注入 bug, 多余字符, **不影响启动** |

### 2.2 多核调度关键参数 — **全部缺失**

| 期望参数 | 实际 | 影响 |
|----------|------|------|
| `isolcpus=` | **未设置** | Linux 默认调度器会使用 **全部 4 个核** |
| `maxcpus=` | **未设置** | SMP 全开, 4 核全部上线 |
| `mem=` | **未设置** | Linux 可用物理内存无限制 (实际被设备树自然划分) |
| `memmap=` | **未设置** | 无显式保留区语法 (使用设备树 `reserved-memory`) |
| `nohz_full=` | **未设置** | 4 核都有 timer tick, 无 adaptive-ticks 模式 |
| `rcu_nocbs=` | **未设置** | 所有核都跑 RCU callback |

### 2.3 `/boot` 目录 — 镜像特性

```
drwxr-xr-x 2 root root 4096 /boot
(empty!)
```

- **没有 `/boot/extlinux.conf`**
- **没有 `/boot/uEnv.txt`**
- **没有 `/boot/grub/`**

PIOS 2.2 镜像采用 U-Boot + fitImage 模式: 镜像直接烧写到 **SD 卡 4 MiB 偏移处**, 不走本地 boot 脚本。

### 2.4 内核版本与启动流程

```
Linux 6.6.63-phytium-embedded-v3.2 #1 SMP PREEMPT Wed Jun 25 11:38:56 CST 2025 aarch64
```

**完整启动链路**:
```
SD 卡 4 MiB 偏移
  └─ fitImage (kernel.gz + board_amp.dtb)
       └─ U-Boot 加载
            ├─ 解压 kernel.gz → 0x80080000
            ├─ 加载 board_amp.dtb
            └─ 跳转执行
                 └─ Linux kernel init
                      ├─ 解析 dtb 中的 /reserved-memory (409 MiB 标记为 rproc 保留)
                      ├─ SMP bringup secondary CPUs (CPU1/2/3)
                      ├─ 注册 remoteproc0 (homo_rproc@0)
                      ├─ 探测 virtio_rpmsg_bus 驱动
                      └─ 创建 /dev/remoteproc0 (cdev 248:0)
                           └─ 用户态 echo start > state → 启动 FreeRTOS
                                └─ CPU1 被 remoteproc "领走", Linux 不再调度 CPU1
```

### 4.2 源码层审计 — 多核数据交换关键模块

#### 4.2.1 SHM Ring Buffer (`freertos/inc/shm_data.h` + `task2_freertos/shm_reader.c`)

**核心数据结构** (每节点一个独立 region):
```c
typedef struct {            // 偏移
    volatile uint32_t wi;       // 0   write index (FreeRTOS 单写)
    volatile uint32_t ri;       // 4   read index  (Linux 单写)
    volatile uint32_t count;    // 8   帧计数      (FreeRTOS 单写)
    volatile uint32_t frame_sz; // 12  帧大小      (init 时定)
    uint8_t data[];             // 16  ring buffer
} ShmRegion;
```

**Producer 写入逻辑 (shm_put_frame)**:
```c
u32 wi = SHM_WI(base);
u32 ri = SHM_RI(base);
u32 next = (wi + frame_size) % data_size;
if (/* 会覆盖未读 */) return;     // 满则丢帧
for (i=0; i<frame_size; i++)      // ★ 字节级 volatile 写
    dst[(wi+i) % data_size] = frame[i];
SHM_WI(base) = next;               // ★ 无 release barrier
SHM_CNT(base)++;
```

**Consumer 读取逻辑 (shm_reader.c main)**:
```c
while (shm->count == 0) usleep(10000);
ri = (shm->wi - frame_sz + data_size) % data_size;
while (1) {
    u32 cur_count = shm->count;   // ★ 无 acquire barrier
    while (start_count <= cur_count) {
        for (i=0; i<frame_sz; i++) // 字节级读
            fbuf[i] = shm->data[(ri+i) % data_size];
        fwrite(fbuf, frame_sz, 1, stdout);
        ri = (ri + frame_sz) % data_size;
        start_count++;
    }
}
```

#### 4.2.2 SPSC / 无锁队列 / 内存屏障关键字检索

```
grep -E "SPSC|spsc|lock[_ -]?free|memory_order|atomic|__sync|__atomic|__dmb|__dsb|DMB|DSB" \
     --include="*.c" --include="*.h" freertos/ task2_linux/ task2_freertos/
```

| 关键字 | 命中 | 说明 |
|--------|------|------|
| `SPSC` / `spsc` / `lock-free` | **0 命中** | 没有无锁队列实现 |
| `memory_order` / `atomic` / `__sync` / `__atomic` | **0 命中** | 无 C11 原子操作 |
| `__dmb` / `__dsb` / `DMB` / `DSB` | **0 命中** (lora_uart.c 里有 3 个, 与本次无关) | **SHM 区无内存屏障** |
| `mutex` / `spinlock` / `irqsave` | **0 命中** (除了 shm_print.c 中的日志互斥) | SHM 无锁 |
| `xSemaphore*` | 1 处 (shm_print.c) | 仅日志互斥 |

**结论: 当前 SHM 环形缓冲是 SPSC 模式 (单生产者 FreeRTOS, 单消费者 Linux), 但**:
- **没有显式内存屏障** (rely on `volatile` + AArch64 SoC 硬件一致性)
- **没有原子操作** (rely on 32-bit 对齐 + 硬件自动原子)
- **字节级循环读写** (比 `memcpy` 慢, 且 store buffer 重排窗口更大)

### 4.3 SPSC 风险评估 (Phase 3 视角)

| 风险 | 当前状态 | Phase 3 风险 |
|------|---------|--------------|
| 单写单读 (SPSC) 数据竞争 | **OK** (AArch64 SoC 强一致) | **MPSC 会出现** (同节点多消费者) |
| 内存重排导致撕裂 | **未观察到** (硬件兜底) | 高并发下可能偶发 |
| 字节级 volatile 写性能 | 实测 1804 fps 5bus | 8x 字节级 → 9bus 1394 fps 还能接受 |
| Ring buffer 容量 32KB / 64KB | 5bus 32K@64B = 512 帧 ≈ 256ms | 高并发跑 8 个同节点实例会撞满 |
| 写入与覆盖保护 | 满则丢帧 (silent drop) | 高并发需要"丢帧计数"上报 |
| Linux `mmap O_SYNC` | 是 | 写入有 sync, 但读不强制 refresh cache |

## 总结: Phase 3 实施前需要修正的认知

| 误判 | 修正 | 影响 |
|------|------|------|
| 4 GB 内存 | **8 GB** | Phase 3 高并发可分配更多 SHM |
| CPU3 不可用 | **可用, 是 A76 big 核 (1.8 GHz)** | 5bus/9bus 应迁到 CPU3, 释放 CPU0 给桌面 |
| CPU1 是 big 核 | **A55 小核** | SIM 任务计算量小, 不影响 |
| 无 SPSC 实现 | **没有显式 SPSC, rely on SoC 硬件一致性** | Phase 3 需加 `__atomic_store_n(__ATOMIC_RELEASE)` + `__ATOMIC_ACQUIRE` |
| SHM 字节级读写 | **字节级 volatile** | 性能差但安全, 暂不优化 |
| 桌面进程占 4 核 | **必须先关闭 firefox/Xorg/xfce4** | Phase 3 启动脚本要停桌面 |
| Dashboard 单进程 Flask | **需要 gunicorn + 多 worker** | 报告 WebServer 高并发改造点 (Phase 3.4) |
| `cma=256m` 占用 256 MiB | 是 | Linux 可用 ~7 GiB, 不影响 |
